# ADR-005: Dual-runtime cleaner with parity fixtures

**Date:** 2026-07-03 **Status:** Accepted **Deciders:** Aditya Kumar Darak

## Context

The same cleaning logic runs in two places: the Python CLI (`src/linkedin_analyzer/core/text.py` and the cleaners around it) and the web app (`web/src/field-cleaners.js`, composed by `web/src/cleaner.js`). The web app cannot call Python (it runs fully in the browser), and the CLI cannot depend on the browser bundle. Keeping two independent implementations invites silent drift: a fix applied to one runtime but not the other would hand users different Excel output from the same export.

## Decision

Maintain the JavaScript cleaner as a deliberate port of the Python one, and pin the two together with cross-runtime parity tests that both `make test` runs:

- Hand-written `tests/fixtures/*-parity.csv` fixtures pin exact expected values for readable, targeted cases (quote and backslash escaping, formula-injection prefixes, control characters, smart quotes, impossible dates).
- A seeded synthetic corpus (`tests/fixtures/*-corpus.csv`, generated by `make gen-parity-corpus`) drives a few hundred rows per type through both cleaners. Both `tests/test_web_parity.py` and `web/tests/parity.test.js` assert their cleaned output equals one checked-in expected file (`tests/fixtures/parity-corpus-expected.json`).

## Consequences

- A cleaning-behavior change in only one runtime fails CI, so the two surfaces cannot silently diverge.
- The cost is a real second implementation plus the discipline to regenerate the corpus after an intentional cleaning change. The generator keeps date payloads timezone independent so the expected file is stable across machines.
